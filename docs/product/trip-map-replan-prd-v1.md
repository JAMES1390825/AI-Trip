# Trip Canvas 新功能 PRD（完整版 v1.0）
更新时间：2026-03-19  
文档状态：Scope Locked (For Implementation)  
适用范围：`apps/web-client` + `apps/trip-api-go`

---

## 1. 背景与问题

当前产品已经具备基础的“生成行程 -> 重排 -> 保存”能力，但在执行层体验仍有关键缺口。

- 地图展示尚未真正落地，`/trip` 页面当前为占位态，无法在地图上查看点位与路线。
- 重排能力是“可用但不够可控”，用户无法锁定已确认活动，也不能精确只调某个时间窗口。
- 重排结果缺少结构化解释，用户无法快速理解“改了什么、为什么改”。
- 版本管理偏弱，用户难以对比不同重排结果，也不便回退到历史版本。

---

## 2. 现状审计（代码事实）

- 前端行程详情页地图区域仍是占位文案，未渲染真实地图。
- 前端已具备地图点位预处理函数，存在可复用基础能力。
- 后端行程数据已包含地图所需基础字段（`poi_lat`、`poi_lon`、`navigation_url`、`transit_legs`、`map_provider`）。
- 后端已支持通用重排接口，但未将“锁定活动 + 局部窗口重排 + 变更清单”做成完整协议。

---

## 3. 产品目标

### 3.1 业务目标

- 将行程详情页升级为“可执行地图视图”，提升行程落地能力。
- 将重排升级为“可控编辑能力”，降低用户对被误改的焦虑。
- 将版本与差异能力产品化，提升用户对系统可解释性与信任度。

### 3.2 体验目标

- 用户在 `/trip` 页可直接看地图、看路线、点时间线联动定位。
- 用户可以锁定关键活动，并只重排指定天/时段。
- 每次重排后可见结构化变更摘要，并可一键回退上一版本。

### 3.3 指标目标（上线后 2 周观测）

- `/trip` 页地图可用率 >= 99.5%（非空白、非崩溃）。
- 重排后立即保存率提升 >= 12%。
- “重排后再次立即重排”比例下降 >= 20%。
- 用户因“行程被改乱”触发的反馈下降 >= 30%。

---

## 4. 非目标

- 本期不做多人实时协作编辑。
- 本期不做跨城市多段联程规划。
- 本期不接第三方实时票务交易闭环。
- 本期不做离线地图与离线导航。

---

## 5. 用户与场景

### 5.1 目标用户

- 已生成初版行程、需要二次优化的自由行用户。
- 有部分刚性约束（已预约景点/餐厅）的用户。
- 需要在出行中快速执行与临时调整的用户。

### 5.2 关键场景

- 出发前一天，用户锁定已预订项目并局部优化剩余时段。
- 旅行中，用户临时替换景点，要求其他安排尽量不动。
- 用户对比两个版本后，回退到更满意的一版继续执行。

## 6. 功能范围（完整版）

### 6.1 模块 A：地图执行视图（Map Execution View）

- 在 `/trip` 页面接入真实地图，替换占位内容。
- 基于 `days.blocks` 渲染 POI 标记点（Marker）。
- 按每日行程顺序绘制路线（Polyline）。
- 支持“全部天数 / 单天”切换。
- 支持“时间线点击 -> 地图定位与高亮”。
- 支持“地图点击 -> 时间线滚动并高亮对应 block”。
- 保留并强化外部导航能力（`navigation_url`）。

### 6.2 模块 B：锁定活动（Lock/Unlock）

- 用户可对 block 执行锁定/解锁。
- 锁定后任何重排不得修改该 block 的 POI、时间、顺序。
- 被锁定 block 在 UI 上有明确状态标识。

### 6.3 模块 C：局部窗口重排（Window Replan）

- 支持按“天 + 时间窗口”重排。
- 仅重排窗口内且未锁定的 block。
- 窗口外内容必须保持不变。
- 若窗口不可重排，返回可读冲突信息。

### 6.4 模块 D：变更解释与版本管理

- 每次重排返回结构化变更清单 `changes`。
- 提供“本次改动摘要”面板。
- 行程引入 `version` 机制并支持回退。
- 支持对比“当前版本 vs 上一版本”的关键差异。

## 7. 交互设计要求

### 7.1 `/trip` 页面布局

- 左侧主区域：地图视图（桌面优先）。
- 右侧区域：时间线 + 锁定按钮 + 局部重排入口。
- 底部区域：变更摘要、交通段、风险提醒。

### 7.2 核心交互

- 点击时间线 block：地图聚焦到对应 marker，弹出点位摘要。
- 点击 marker：时间线滚动到 block 并高亮。
- 锁定按钮：点击后即时状态反馈，失败给出原因。
- 局部重排提交后：先显示 loading，再展示变化清单。

### 7.3 空态与异常态

- 无坐标数据时展示可读空态，不影响时间线使用。
- 地图 SDK 失败时降级到“时间线 + 外链导航”。
- 重排失败需给出可执行下一步，不允许仅提示“失败”。

## 8. 数据模型变更

### 8.1 `block` 新增字段

- `block_id: string`（稳定标识，用于联动与版本 diff）
- `locked: boolean`（是否锁定）
- `lock_reason: string`（可选，便于后续解释）

### 8.2 `itinerary` 新增字段

- `version: number`（从 1 递增）
- `parent_version: number | null`（来源版本）
- `changes: ChangeItem[]`（本次变更清单）
- `conflicts: ConflictItem[]`（冲突信息）

### 8.3 结构定义（逻辑）

- `ChangeItem = { day_index, block_id, start_hour, old_poi, new_poi, reason, change_type }`
- `ConflictItem = { code, message, day_index, block_id? }`

## 9. 接口设计（基于现有 API 扩展）

接口字段级契约与示例请参考：[trip-map-replan-api-contract-v1.md](./trip-map-replan-api-contract-v1.md)


### 9.1 重排接口

`POST /api/v1/plans/replan`

请求体新增约定：

- `patch.change_type: "lock" | "unlock" | "replan_window" | "budget" | "preferences" | "poi" | "date"`
- `patch.targets: [{ day_index, block_id, start_hour, end_hour }]`
- `patch.keep_locked: true`（默认 true）
- `patch.replan_scope: "day" | "window" | "global"`（默认 `window`）

响应体新增约定：

- `version`
- `parent_version`
- `changes`
- `conflicts`
- `warnings`（保留）

### 9.2 版本回退接口

`POST /api/v1/plans/revert`

请求体：

- `current_itinerary`
- `target_version`

响应体：

- `itinerary`（回退后的最新版本对象）
- `version`
- `changes`（标记为 `revert` 类型）

### 9.3 版本历史接口

`GET /api/v1/plans/saved/{id}/versions?limit=20`

响应体：

- `[{ version, created_at, summary, change_count }]`

## 10. 业务规则

- 锁定 block 不可被任何重排类型改动。
- 局部重排必须严格限制在目标窗口内。
- 当候选 POI 不足时，允许保留原安排并写入 `warnings`。
- 当窗口内全为锁定 block 时，返回 `conflicts`，不做改动。
- 任一请求仅允许操作当前用户自己的行程。
- 任一失败必须返回明确 `error code + message`。

## 11. 非功能要求

- 地图首屏渲染 P95 < 1.2s（已有 itinerary 情况）。
- 重排接口 P95 < 800ms（规则模式）。
- 关键交互（点击 block 定位地图）响应 < 200ms。
- 主要页面在桌面与移动端均可完整使用。
- 所有新增字段向后兼容，不破坏旧客户端。

## 13. 验收标准（UAT）

- 打开 `/trip` 即可见可交互地图，不再出现“接入中”占位文案。
- 点击任意 block，地图定位并高亮对应点位。
- 锁定后执行重排，锁定 block 内容保持不变。
- 仅重排第 N 天下午时，其他天数据完全不变。
- 响应中 `changes` 与真实变化一致。
- 可查看版本列表，并能成功回退到指定版本。
- 异常场景有明确用户提示与可执行下一步。

## 15. 风险与应对

- 风险：地图 SDK key 管理缺失导致环境不一致。  
应对：补齐 `.env.example` 与启动校验。

- 风险：局部重排规则复杂度上升导致边界缺陷。  
应对：先固化重排窗口规则并补充表驱动测试。

- 风险：变更解释文案不一致影响理解。  
应对：统一 `reason code -> 文案模板` 映射。

- 风险：版本历史过多引发存储膨胀。  
应对：先限制每个计划保留最近 20 个版本。

---

## 16. 已确认决策（2026-03-19）

- 地图供应商：本期固定高德（Amap），不做多 Provider 抽象。
- 局部重排窗口：支持任意小时段，不限制为“上午/下午/晚上”快捷模式。
- 回退策略：采用“生成新版本回退结果”，不直接覆盖当前版本。
- 历史版本保留策略：每个行程保留最近 20 个版本。
