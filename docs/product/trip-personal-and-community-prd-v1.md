## 1. 文档目标

本文件定义 AI Trip 在“真实数据规划”之上的下一阶段产品主线：

`个人化持续学习 + 社区化公开经验`

本文件重点回答 6 个问题：

1. 当前产品为什么需要把“个人层”和“社区层”一起纳入新需求。
2. 个人层应该默认提供哪些能力，哪些必须仍然保持私有。
3. 社区层应该先做哪些能力，哪些现在不适合做太重。
4. 个人信号、社区信号与地图事实之间的边界怎么划。
5. 现有双 Tab 产品结构下，社区能力应如何接入，不破坏当前主流程。
6. 这条新主线应该如何分阶段落地。

核心结论：

`AI Trip 后续不应只做“生成更像 AI 的路线”，而应演进为“越用越懂你、也能借到别人公开经验”的旅行决策系统。`

### 1.1 当前实现快照

截至 2026-04-02，仓库内已完成本 PRD 的第一批主线能力：

1. 用户私有画像摘要与排序底座已接入 Go 后端。
2. 社区帖子发布 / 列表 / 投票 API 已落地。
3. 移动端 `行程` Tab 已接入社区分享发布与社区精选浏览。
4. 移动端 `AI规划` Tab 已接入目的地社区参考选择。
5. `generate-v2` 已支持 `community_post_ids`，并将公开社区帖子作为软信号参与候选排序。
6. 已支持从已保存行程一键生成社区分享草稿。

当前仍待深化的部分主要是：

1. 社区信号实验、收益评估与参数调优。
2. 更完整的后台治理工具与运营视图。

## 2. 背景与问题

### 2.1 当前阶段已经解决的问题

当前主线已经开始解决：

1. 正式点位需要真实来源。
2. 路线、天气、校验结果不能主要靠模型脑补。
3. 移动端主流程应围绕“AI 规划 -> 地图执行 -> 调整 -> 保存”收口。

### 2.2 当前阶段仍然存在的问题

即使真实数据链路逐步成立，产品仍有 4 个明显缺口：

1. 同一个用户多次使用后，系统仍然不够“记住你”。
2. 优秀路线经验只存在于单次结果里，无法沉淀为可复用资产。
3. 用户之间的公开经验尚未进入系统，导致路线仍偏“系统产物”，不够“有人味”。
4. 如果未来只靠模型继续加 prompt，优化空间会越来越有限，且不可控。

### 2.3 为什么个人层和社区层要一起设计

因为这两层分别解决不同问题：

1. 个人层解决“更像我”。
2. 社区层解决“更像人走出来的经验”。

如果只做个人层，系统会越来越懂单个用户，但无法形成公开经验资产。  
如果只做社区层，系统会有公开路线内容，但对当前用户仍然不够贴身。

因此下一阶段最合理的主线不是二选一，而是：

`个人默认私有学习 + 社区明确公开贡献`

## 3. 产品目标

### 3.1 业务目标

1. 提高 `plan_generated -> plan_saved` 转化。
2. 提高用户复用与回访频率，让用户觉得“下次更省心”。
3. 让优秀路线沉淀为长期可复用资产，而不是单次生成后消失。
4. 为后续社区化增长提供第一批高质量公开内容基础。

### 3.2 用户体验目标

1. 用户会感知到“系统越来越懂我”，但不会觉得隐私被冒犯。
2. 用户能用别人的公开路线做参考，但不会被错误社区信息带偏。
3. 用户在结果页看到的是“真实事实 + 个性化偏好 + 可验证的公开经验”共同形成的路线。

### 3.3 北极星指标

建议新增 4 组核心指标：

1. 个人化收益：
`returning_user plan_saved_rate uplift`

2. 个人层学习质量：
`block_removed_rate decline`  
`block_replaced_rate decline`

3. 社区层资产质量：
`public_route_reference_rate`  
`public_route_save_rate`

4. 社区层治理质量：
`reported_public_route_rate`  
`community_signal_acceptance_rate`

## 4. 范围与非目标

### 4.1 本期范围

本期设计包含两条能力线：

1. 个人层：
- 用户私有行为学习
- 私有画像
- 个性化排序
- 个性化解释

2. 社区层：
- 公开路线发布
- 公开路线参考生成
- 结构化标签
- 有用度投票与举报
- 社区信号聚合进入排序与解释

### 4.2 本期不做

1. 不做泛内容社区 feed。
2. 不做长文本攻略流。
3. 不做实时多人协作。
4. 不做完整创作者体系或商业化分发。
5. 不做社区内容直接覆盖地图、天气、路线真值。

## 5. 核心设计原则

### 5.1 事实优先

地图 POI、路线、天气、营业状态等仍由 provider 事实层提供。  
个人信号和社区信号都只能影响排序、替代建议和解释层。

### 5.2 私有默认开启，公开必须显式授权

个人行为默认进入私有学习层。  
任何要进入社区层的内容，都必须经过用户主动发布或明确授权。

### 5.3 结构化优先于自由文本

社区层第一阶段优先使用：

1. 公开路线
2. 结构化标签
3. 有用度投票

而不是优先做自由评论主导的内容产品。

### 5.4 参考生成，不是直接照抄

社区路线进入用户规划时，推荐模式应是：

`参考公开路线 -> 结合当前用户需求重新生成`

而不是把别人的路线直接塞给当前用户。

### 5.5 不破坏当前双 Tab 主线

当前 App 已收口为 `行程 / AI规划` 双 Tab。  
新需求第一阶段不建议立刻新增独立社区 Tab。

更合理的接入方式是：

1. 在 `AI规划` 中加入“参考公开路线”入口。
2. 在 `行程` 中加入“发布为公开路线”与“查看引用来源”入口。
3. 等社区内容体量与互动足够后，再评估是否需要独立社区发现页。

## 6. 个人层需求

### 6.1 个人层目标

让系统逐步记住用户自己的真实偏好，并只影响该用户自己的下一次规划。

### 6.2 个人层必做能力

1. 用户关键行为事件记录
2. 用户私有画像生成
3. 个性化候选排序
4. 个性化排程软约束
5. 轻量个性化解释
6. 个性化学习开关与清理能力

### 6.3 个人层具体需求

#### FR-PER-01 私有学习默认开启

用户首次使用时，系统默认启用私有行为学习。  
后续需要支持：

1. 关闭个性化学习
2. 清空个人学习记录

#### FR-PER-02 关键行为事件标准化

至少支持以下事件：

1. `plan_generated`
2. `plan_saved`
3. `block_removed`
4. `block_replaced`
5. `block_locked`
6. `poi_detail_opened`
7. `navigation_started`
8. `preference_changed`

#### FR-PER-03 用户私有画像

系统需要持续维护 `UserPrivateProfile`，至少包含：

1. 显式偏好
2. 类别偏好
3. 标签偏好
4. 节奏画像
5. 通勤容忍度
6. 风险偏好
7. 置信度

#### FR-PER-04 个性化排序

用户画像应进入候选排序，但只作为软加权项。

必须满足：

1. `must_go` 高于画像
2. `avoid` 高于画像
3. 冷启动用户完整回退
4. 单一画像因子影响受限

#### FR-PER-05 个性化解释

当置信度足够时，结果页应展示 1~2 条轻量解释，如：

1. “这次更偏步行和江景，是因为你最近更常保留这类点位。”
2. “已尽量减少跨区通勤，因为你最近更常调整长距离移动。”

#### FR-PER-06 用户控制权

用户后续应可看到：

1. 个性化学习是否开启
2. 是否允许系统使用历史行为优化结果
3. 是否清除历史学习记录

第一阶段可以不做完整管理页，但必须在系统设计里预留。

## 7. 社区层需求

### 7.1 社区层目标

让用户的公开路线和公开经验沉淀为结构化公共资产，并安全地进入规划系统。

### 7.2 社区层第一阶段的核心定位

不是做“内容社区”，而是做“公开路线与结构化经验层”。

第一阶段建议优先做：

1. 公开路线
2. 结构化标签
3. 参考生成
4. 有用度投票
5. 举报与治理

### 7.3 社区层具体需求

#### FR-COM-01 公开路线发布

用户可以把一条已保存路线发布为公开路线。

公开前需明确：

1. 是否匿名显示
2. 是否允许他人参考生成
3. 是否允许公开标签被继续投票

公开内容至少包括：

1. 路线标题
2. 城市 / 日期范围 / 天数
3. 路线结构摘要
4. 结构化标签
5. 公开说明摘要

#### FR-COM-02 公开路线浏览

其他用户可以查看公开路线的：

1. 路线概览
2. 按天结构
3. 标签
4. 公开说明
5. 被保存 / 被参考次数

#### FR-COM-03 参考公开路线重新生成

用户不能直接把公开路线当作最终事实结果保存。  
应优先支持：

`以公开路线为参考 -> 结合当前需求重新生成`

系统在重新生成时可读取：

1. 公开路线结构
2. 路线标签
3. 公共受欢迎度

但仍需经过事实层检索、路线校验和当前用户偏好融合。

#### FR-COM-04 结构化标签

用户可给公开路线或公开点位提交结构化标签，例如：

1. `夜景友好`
2. `雨天可行`
3. `拍照出片`
4. `亲子不累`
5. `情侣约会`
6. `citywalk`

第一阶段不建议先开放自由标签泛滥，需有标签白名单或治理机制。

#### FR-COM-05 有用度投票

用户可对公开路线或公开标签投：

1. 有帮助
2. 没帮助

该结果用于：

1. 社区内容排序
2. 标签质量控制
3. 社区信号聚合

#### FR-COM-06 举报与审核

用户必须可以举报：

1. 低质量路线
2. 错误标签
3. 冒犯性内容
4. 不适当公开内容

平台需支持状态流：

`draft -> published -> limited -> reported -> removed`

#### FR-COM-07 社区信号聚合

社区层进入主系统前，需被整理成结构化公共信号，例如：

1. `public_tag_scores`
2. `public_route_mentions`
3. `community_save_rate`
4. `community_reference_rate`

这些信号可进入排序和解释层，但不能直接当事实写回 itinerary。

## 8. 产品体验设计

### 8.1 当前 IA 下的接入方式

第一阶段不建议新增独立社区 Tab。  
建议把新能力嵌入现有双 Tab：

#### AI规划 Tab

新增：

1. “参考公开路线”入口
2. 公开路线卡片推荐
3. 参考来源说明

#### 行程 Tab

新增：

1. “发布为公开路线”
2. “查看我的公开路线”
3. “查看本路线被参考次数”

### 8.2 关键用户流程

#### Flow A：个人学习

```text
用户生成路线
  -> 查看 / 删除 / 替换 / 保存 / 导航
  -> 系统更新个人画像
  -> 下次同城市或相似风格规划时个性化生效
```

#### Flow B：公开路线发布

```text
用户保存路线
  -> 选择公开发布
  -> 补充标题与结构化标签
  -> 提交审核
  -> 发布成功
```

#### Flow C：参考公开路线

```text
用户进入 AI规划
  -> 查看公开路线卡片
  -> 选择“参考生成”
  -> 系统结合个人偏好与公开路线结构重新生成
```

## 9. 数据边界与治理要求

### 9.1 三层边界

| 层 | 默认权限 | 是否可跨用户 | 是否可在线用于当前用户规划 | 说明 |
|---|---|---|---|---|
| 个人层 | 私有 | 否 | 是 | 默认开启，按 `user_id` 隔离 |
| 聚合层 | 平台匿名统计 | 是 | 是 | 必须达样本阈值 |
| 社区层 | 用户显式公开 | 仅公开内容可见 | 是 | 必须经过治理 |

### 9.2 禁止事项

必须明确禁止：

1. 把用户私有原始行为直接提供给其他用户。
2. 把社区自由文本直接写入 itinerary 的事实字段。
3. 让模型根据社区自由文本改写营业时间、天气或路线时长。

### 9.3 样本阈值

社区信号进入排序前，至少应满足：

1. 最小公开路线数阈值
2. 最小不同用户数阈值
3. 最近窗口样本阈值
4. 举报率不过高

## 10. 排序与解释接入规则

### 10.1 排序接入优先级

推荐优先级：

1. 当前用户显式输入
2. 地图 / 路线 / 天气事实约束
3. 用户私有画像
4. 聚合统计
5. 社区结构化信号
6. explain 层文案生成

### 10.2 个人与社区的角色区别

个人层负责：

1. 更像你
2. 更少踩你自己不喜欢的坑
3. 更符合你的节奏

社区层负责：

1. 更像真实用户走出来的路线
2. 提供公共经验标签
3. 给 explain 层更多“有人味”的理由依据

### 10.3 explain 层限制

解释层可以说：

1. “很多公开路线都把这里作为夜景段落。”
2. “你最近更常保留步行和江景路线，所以本次排序做了偏重。”

解释层不能说：

1. “这里今天一定营业到几点。”
2. “社区都说一定别去，所以系统删除了它。”

## 11. 分期建议

### Phase A：个人层正式上线

目标：

- 个性化学习成为默认能力

交付：

1. 个人事件标准化
2. 私有画像
3. 个性化排序
4. 轻量个性化解释

### Phase B：社区层轻量发布

目标：

- 第一批公开路线资产形成

交付：

1. 公开路线发布
2. 结构化标签
3. 公开路线浏览
4. 举报与审核基础

### Phase C：公开路线参考生成

目标：

- 社区经验真正进入规划系统

交付：

1. 公开路线参考生成
2. 社区信号聚合
3. 公开路线排序与有用度投票

### Phase D：社区体验增强

目标：

- 让社区资产可持续增长

交付：

1. 我的公开路线资产页
2. 路线引用与保存统计
3. 更强的标签治理
4. 是否需要独立社区发现页的评估

---

## 12. 验收标准

### 12.1 个人层验收

1. 用户关闭个性化学习后，后续行为不再进入私有画像更新。
2. 同用户重复使用后，删除和替换率出现下降趋势。
3. 冷启动用户结果与当前版本相比无明显负面偏移。

### 12.2 社区层验收

1. 用户可顺利发布公开路线并完成审核流转。
2. 公开路线可被他人浏览与参考生成。
3. 社区结构化标签可进入 explain 与排序，但不改写事实。
4. 举报与治理链路可用。

---

## 13. 当前建议拍板项

为了让这个需求进入正式开发，建议当前先拍板以下 6 件事：

1. 个人层默认开启，社区层默认关闭，必须显式发布。
2. 当前双 Tab 结构不变，社区先不独立成 Tab。
3. 社区第一阶段先做公开路线和结构化标签，不做重内容 feed。
4. 公开路线进入系统时一律走“参考生成”，不直接照搬保存。
5. 个人层优先于社区层影响当前结果。
6. 任何社区信号都不能覆盖 provider 事实层。
