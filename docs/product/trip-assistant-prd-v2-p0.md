## 1. 产品定义

Trip Canvas 的定义升级为：

- 不是“行程生成器”，而是“旅途规划与执行助手”。
- 覆盖用户完整旅行周期：出发前（决策）-> 旅途中（执行）-> 返程后（复盘）。
- 核心价值是：省心、可控、可回退、可协作、可复用。

一句话定位：

> 帮用户从“想去哪”到“玩得好、少踩坑、花钱透明”全链路完成旅行。

---

## 2. 用户与场景

### 2.1 目标用户

- 25-40 岁自由行用户（2-5 天国内游为主）。
- 有明确预算约束、时间约束、同行协同需求。
- 典型痛点：信息分散、计划容易改乱、突发变化处理慢、同行沟通成本高。

### 2.2 核心用户任务（JTBD）

- 出发前：快速拿到可执行多方案并做出决策。
- 旅途中：随时知道“现在做什么、下一步去哪、为什么这样安排”。
- 变化时：在不破坏关键安排的前提下快速重排。
- 返程后：沉淀可复用经验，下次规划更快。

---

## 3. 体验目标（P0）

### 3.1 业务目标

- 从“单方案生成”升级为“可比较、可执行、可协作”的 MVP。
- 明显降低用户在执行阶段的困惑和沟通成本。

### 3.2 用户体验目标

- 用户 10 分钟内完成：需求补全 -> 方案对比 -> 选定计划。
- 用户在旅途中 30 秒内完成：定位当前任务 -> 查看导航 -> 必要时窗口重排。
- 用户每次关键操作都可追溯（版本）且可撤销（回退）。

### 3.3 量化指标（上线后 2 周）

- `plan_generated -> plan_saved` 转化率提升 >= 20%。
- 重排后 10 分钟内二次重排比例下降 >= 20%。
- 地图可用率 >= 99.5%。
- 分享链接使用率 >= 25%（保存行程用户中）。
- 关键流程失败可恢复率（有明确下一步）>= 95%。

---

## 4. 全量能力版图（分期）

### 4.1 全量能力（长期）

- A 出发前：灵感推荐、约束采集、多方案对比、预算拆解、预订清单。
- B 旅途中：今日视图、地图执行、动态重排、风险预警、进度打卡、记账。
- C 返程后：复盘报告、账单分摊、回忆轴、模板复用。
- D 横向：可解释性、可协作、可回退、性能与稳定、观测与实验。

### 4.2 本文范围（P0）

只做最影响主链路的 7 个模块：

1. 多方案生成与对比
2. 出发前准备清单（最小可用）
3. 今日执行视图
4. 动态重排（锁定 + 窗口）统一交互
5. 版本差异可视化 + 回退确认
6. 行程分享（只读）
7. 地图与风险诊断（可恢复）

## 5. 详细需求（P0）

## 5.1 模块 M1：多方案生成与对比

### 用户故事

- 作为用户，我希望一次看到 2-3 套不同风格方案，以便快速比较而非反复重生成。

### 功能需求

- 生成后返回 `economy / balanced / experience` 三套方案（可配置至少两套）。
- 每套展示：总预算、日均通勤时长、打卡密度、风险数、亮点摘要。
- 支持“设为当前方案”并进入 `/trip` 执行页。

### 交互要求

- 对比卡片必须同屏展示核心指标，不允许用户进入详情后才看到差异。
- 对比支持“仅看差异”切换。

### 验收标准

- 用户在不返回对话页的前提下可完成方案选择。
- 方案切换后可保存，保存记录保留方案标签。

---

## 5.2 模块 M2：出发前准备清单（最小可用）

### 用户故事

- 作为用户，我希望知道哪些项目要提前预约和准备，避免临近出发遗漏。

### 功能需求

- 自动生成准备清单分类：证件、预约、交通、住宿、行李、支付。
- 每项支持状态：`todo / done / skipped`。
- 支持“最晚处理时间”字段。

### 交互要求

- 在 `/trip` 页顶部提供“出发前清单”入口（桌面）与快捷卡片（移动）。

### 验收标准

- 用户可在清单中完成状态切换并持久化。
- 清单状态不影响行程版本号。

---

## 5.3 模块 M3：今日执行视图

### 用户故事

- 作为旅途中用户，我只想看“今天接下来干什么”，不想在全行程中来回找。

### 功能需求

- 在 `/trip` 新增“今天”模式：只展示当天时间线与地图点位。
- 高亮“当前时段”和“下一时段”。
- 显示“距下一时段还有 xx 分钟”。

### 交互要求

- 默认进入全量视图，提供 `全部 / 今天` 分段切换。
- 切到今天模式后地图自动应用当天过滤。

### 验收标准

- 当前时段判定准确（基于本地时间）。
- 今天模式下重排仅默认当天，不影响其他天。

---

## 5.4 模块 M4：动态重排统一交互

### 用户故事

- 作为用户，我希望“改一点点”时只影响目标窗口，并且已确认项目不被改乱。

### 功能需求

- 保留并强化现有能力：
- block 锁定/解锁
- 按天 + 小时窗口重排
- 快捷重排（预算/偏好/换点/改日期）

- 统一重排入口：
- “快速优化”与“窗口重排”从视觉上明确区分作用范围。
- 提交前展示影响范围摘要（天数、时段、是否保护锁定）。

### 交互要求

- 每次重排完成后必须展示：`changes + conflicts + warnings`。

### 验收标准

- 锁定块在任何重排类型下保持不变。
- 窗口外块在窗口重排后保持不变。

---

## 5.5 模块 M5：版本差异可视化 + 回退确认

### 用户故事

- 作为用户，我希望在回退前先看懂“要回去的版本和当前版差在哪里”。

### 功能需求

- 在版本列表中新增“查看差异”按钮。
- 差异按 `day -> block` 展示：时间、POI、锁定状态、预算变化。
- 回退动作增加二次确认弹层，展示将回退到的版本摘要。

### 交互要求

- 当前版本与目标版本差异必须支持按天过滤。

### 验收标准

- 用户可在不离开 `/trip` 的情况下完成“对比 -> 回退”。
- 回退后仍遵循“生成新版本”策略，版本链连续。

---

## 5.6 模块 M6：行程分享（只读）

### 用户故事

- 作为组织者，我希望把行程发给同行人查看并评论，不要让所有人都能改动。

### 功能需求

- 保存行程后可生成只读分享链接。
- 访客可查看：时间线、地图、交通建议、提醒。
- 访客可留言评论（可选，P0 可先做只读不评论）。

### 权限规则

- 分享链接仅允许读，不允许重排/回退/删除。
- 链接支持失效（手动关闭）。

### 验收标准

- 未登录访客可打开只读页面。
- 主账号可随时关闭分享访问。

---

## 5.7 模块 M7：地图与风险诊断（可恢复）

### 用户故事

- 作为用户，我希望地图出问题时系统告诉我具体怎么修，而不是只提示失败。

### 功能需求

- 地图状态卡：
- Key 未配置
- SDK 加载失败
- 可能白名单不匹配
- 坐标缺失

- 风险诊断卡：
- 目的地与坐标城市疑似不一致
- 当天通勤时长过长
- 窗口重排冲突原因分级

- 提供“一键恢复动作”：去设置页、重试加载、切换纯时间线模式。

### 验收标准

- 所有失败态均给出可执行下一步。
- 用户可在失败态下继续使用时间线与外链导航。

## 11. 风险与应对

- 风险：多方案生成增加响应时延。  
应对：先用规则模板并行生成，超时降级到 2 方案。

- 风险：分享链接带来隐私风险。  
应对：token 随机且可失效，默认不含用户敏感字段。

- 风险：诊断误报影响信任。  
应对：诊断分级（提醒/警告），并给出“忽略本次”选项。

## 14. 指标与验收阈值（Step 2 评审稿）

本节用于定义 P0 的量化成败标准，避免“功能上线但效果不可判断”。

### 14.1 指标分层

1. 发布闸门指标（必须达标，否则不建议全量发布）
2. 上线后 2 周业务指标（用于评估 P0 是否有效）
3. 质量守护指标（用于防止体验回退）

### 14.2 发布闸门指标（Go/No-Go）

| 类别 | 指标 | 阈值（P0） | 说明 |
|---|---|---|---|
| 功能正确性 | P0 模块验收用例通过率 | 100% | 7 个模块核心用例全部通过 |
| 接口稳定性 | 关键 API 错误率（5xx） | < 0.5% | `/generate` `/replan` `/revert` `/versions` |
| 地图可用性 | 地图渲染成功率 | >= 99.0% | 有坐标且配置有效前提下 |
| 关键交互性能 | `/trip` 首屏可交互 P95 | < 1.5s | 已有 itinerary 场景 |
| 重排响应 | `POST /plans/replan` P95 | < 800ms | 规则模式 |
| 失败可恢复 | 失败态可执行下一步覆盖率 | >= 95% | 有明确修复动作 |
| 移动端可用性 | 关键页面阻断级 UI 缺陷 | 0 个 | iOS/Android 主流分辨率 |

### 14.3 上线后 2 周业务指标（效果判断）

| 指标 | 口径 | 目标 |
|---|---|---|
| 保存转化率 | `plan_saved / plan_generated` | 较基线提升 >= 20% |
| 重排稳定性 | 10 分钟内二次重排率 | 较基线下降 >= 20% |
| 地图健康度 | 地图失败会话占比 | 较基线下降 >= 30% |
| 分享渗透 | 含分享操作的保存行程占比 | >= 25% |
| 版本使用度 | 查看差异或回退会话占比 | >= 20% |
| 今日执行使用度 | 使用“今天模式”会话占比 | >= 35% |

### 14.4 质量守护指标（防回退）

- 崩溃率：前端 JS 致命错误会话占比 < 0.2%。
- 慢接口占比：关键接口 P99 > 2s 的请求占比 < 3%。
- 数据一致性：目的地与坐标不一致诊断率 < 1%。
- 版本链完整性：回退后版本链断裂率 = 0。

### 14.5 埋点口径（新增/沿用）

P0 必须保证以下事件可用：

- `plan_generated`
- `plan_saved`
- `replan_triggered`
- `replan_applied`
- `plan_replanned_window`
- `plan_diff_viewed`
- `plan_reverted`
- `trip_mode_switched`
- `plan_share_created`
- `plan_share_closed`
- `map_diagnostic_seen`
- `risk_diagnostic_seen`

口径约束：

- 每个事件必须带 `user_id`、`session_id`、`plan_id`（若有）、`timestamp`。
- 所有比率指标分母必须采用“有效会话”定义，避免机器人与无效请求污染。

### 14.6 基线采集要求

- Step 2 确认后先做 3-5 天基线采集，再进入灰度。
- 基线窗口内冻结核心算法参数，避免指标波动不可解释。
- 统计看板至少包含：日趋势、分端（桌面/移动）、分入口（home/plan/trip）。

满足以下条件即视为第 2 步完成：

2. 2 周业务指标目标确认。

## 16. API 评审结论（Step 4）

Step 4 以 `trip-assistant-p0-api-delta-v1.1.md` 作为联调冻结基线，核心结论如下：

- 冻结 A：`POST /plans/generate` 响应形状分支
- 不传 `variants`：保持旧单 itinerary 响应
- 传 `variants`：统一返回 `{ plans, degraded }`

- 冻结 B：`diff` 参数命名统一为 `from_version / to_version`
- 冻结 C：分享创建返回 `share_path`，不返回绝对 `share_url`
- 冻结 D：诊断归属拆分
- 后端只返回数据侧 diagnostics
- 前端本地处理地图 key/sdk/白名单问题

### 17.5 N1 详细提案（草稿）

#### 17.5.1 需求目标

- 让“分享链接接收方”在不登录的情况下，也能看到行程地图点位与顺序，提升同行沟通效率。
- 与当前只读分享页（时间线 + 交通 + 风险）形成完整执行视图。

#### 17.5.2 P1 最小范围

- 在 `/share/:token` 页面增加只读地图渲染（高德）。
- 地图支持：
- 点位标注（按时段顺序编号）
- 按天筛选（全部 / 第 N 天）
- 点击时间线项可定位地图点位

- 保持只读约束：
- 不提供锁定、重排、回退、编辑入口
- 仅允许外链导航（高德）

#### 17.5.3 明确不做（本期排除）

- 分享页内重排/评论/协作编辑。
- 访客个性化偏好保存。
- 多地图服务商切换（继续固定高德）。

#### 17.5.4 接口与数据影响

- 默认不新增 API 路由。
- 复用 `GET /api/v1/share/{token}` 返回的 `itinerary.days[].blocks[].poi_coords`。
- 若坐标不足：前端降级为时间线模式并展示地图诊断。

#### 17.5.5 验收标准

- 未登录访问有效分享链接，地图可渲染率 >= 99%（有坐标且 key 配置正确前提下）。
- 访客点击时间线项时，地图能定位对应点位。
- 失效/过期 token 仍保持明确失败态，不泄露敏感信息。

#### 17.5.6 埋点建议

- `share_map_viewed`
- `share_map_day_filtered`
- `share_map_point_focused`
- `share_map_fallback_timeline`


### 17.6 N2 详细提案（草稿）

#### 17.6.1 需求目标

- 把“风险提醒”从只读提示升级为“可执行下一步”，降低旅途中临时改计划成本。
- 重点覆盖高频风险：闭馆、营业时间冲突、预约截止临近。

#### 17.6.2 P1 最小范围

- 风险类型扩展（后端数据诊断 + 前端动作入口）：
- `POI_CLOSED_ON_DATE`：目标日期闭馆
- `POI_OPEN_HOURS_MISMATCH`：时段不在营业时间内
- `APPOINTMENT_DEADLINE_SOON`：预约截止临近（例如 <= 48h）

- 每条风险必须带至少一个动作：
- “替换该时段”
- “加入出发前清单”
- “查看官方信息/预约链接”（若可用）

#### 17.6.3 明确不做（本期排除）

- 实时第三方票务库存接入。
- 所有景区全量营业时间爬取与自动更新。
- 复杂多条件优化引擎（仅做规则触发）。

#### 17.6.4 接口与数据影响

- 在 `itinerary.diagnostics[]` 上做向后兼容扩展，不新增强制字段：
- `code`、`level`、`message`（已有）
- `action`（新增，可选）
- `target`（新增，可选：day/block）

- 不改动现有 `replan` 契约；动作触发仍走既有重排接口。

#### 17.6.5 验收标准

- 风险列表中，P1 范围内风险项“可执行动作覆盖率” >= 95%。
- 用户点击风险动作后 2 分钟内可完成一次有效调整（替换/加入清单/打开外链）。
- 风险失败态必须有降级路径（至少保留时间线可执行）。

#### 17.6.6 埋点建议

- `risk_diagnostic_seen`
- `risk_diagnostic_action_clicked`
- `risk_diagnostic_resolved`

### 17.7 N3 详细提案（草稿）

#### 17.7.1 需求目标

- 降低“出发前忘记处理关键事项”的概率，提高清单完成率。
- 让清单从“被动查看”变成“主动提醒”。

#### 17.7.2 P1 最小范围

- 基于 `due_at` 与 `start_date` 的提醒策略：
- 固定提醒窗口：`T-7 / T-3 / T-1`
- 若任务本身有 `due_at`，以 `due_at` 优先

- 提醒触达形态：
- 应用内提醒条（首页/行程页）
- 浏览器通知（用户开启后生效）

- 清单视图支持“一键只看未完成 + 即将到期”。

#### 17.7.3 明确不做（本期排除）

- 手机系统推送（APNs/FCM）。
- 第三方日历双向同步（Google/Outlook/日历订阅）。
- 多人协同任务分派。

#### 17.7.4 接口与数据影响

- 后端接口不强制变更；先复用 `GET/PUT /tasks`。
- 前端可增加本地提醒元数据，不影响现有任务字段兼容。
- 若进入 P2，再评估新增 `reminder_at` 服务端字段。

#### 17.7.5 验收标准

- 开启提醒用户中，出发前任务完成率较基线提升 >= 20%。
- 即将到期任务在 24 小时内可见率 >= 99%（应用内）。
- 关闭提醒后不再触发浏览器通知。

#### 17.7.6 埋点建议

- `pretrip_reminder_scheduled`
- `pretrip_reminder_shown`
- `pretrip_reminder_clicked`

### 17.8 N4 详细提案（草稿）

#### 17.8.1 需求目标

- 在“今天模式”中提供明确执行反馈，让用户知道“我已经做完了什么、下一步做什么”。
- 减少旅途中来回滚动查找与决策疲劳。

#### 17.8.2 P1 最小范围

- 时段执行状态：
- `pending / done / skipped`
- 支持一键标记完成、跳过

- 自动推进：
- 完成当前项后自动聚焦下一待执行项
- 顶部显示“今日进度：x/y”

- 地图联动：
- 当前待执行项自动高亮对应点位（可手动切换）

#### 17.8.3 明确不做（本期排除）

- 位置感知自动打卡。
- 跨设备实时同步冲突解决。
- 团队成员执行态共享。

#### 17.8.4 接口与数据影响

- P1 可先本地持久化（不阻塞上线）。
- 若进入稳定期，再评估新增 `execution_state` 持久化接口（`v1.2.x`）。
- 不影响现有行程版本链策略。

#### 17.8.5 验收标准

- 今天模式用户中，完成/跳过操作成功率 >= 99%。
- 完成动作后自动聚焦下一项成功率 >= 95%。
- 用户在今天模式下平均定位下一项时间下降 >= 30%。

#### 17.8.6 埋点建议

- `trip_today_block_done`
- `trip_today_block_skipped`
- `trip_today_next_focused`
