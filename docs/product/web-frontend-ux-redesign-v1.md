# Web 前端体验改版 PRD（v1.1 可执行版）

更新时间：2026-03-12  
文档状态：Approved for Build  
适用范围：Web 前台 + 独立后台控制台（前后台分离）

---

## 1. 背景与问题

现有 Web 原型功能链路可跑通，但用户体验存在明显工程化问题：

1. 单页承载过多任务，信息密度过高，用户无法快速理解“第一步做什么”。
2. 用户视图混入工程信息（参数、调试项、状态字段），破坏产品感。
3. 页面结构偏“工具台”，缺少旅行产品的内容层次与节奏。
4. 技术字段和枚举值泄露到用户文案，影响可读性与信任感。

---

## 2. 产品目标（本次改版）

1. 将 Web 前台改造成用户可直接使用的产品界面，不再像工程面板。
2. 前后台彻底分离：用户端不出现开发者模式、参数配置、调试输出。
3. 多页面结构承载用户旅程：发现 -> 对话规划 -> 行程执行 -> 历史复用。
4. 行程详情页以地图为主视图，时间线联动为辅。
5. 不向用户展示置信度数值或内部模型状态。

---

## 3. 已确认决策（冻结）

1. 用户端与后台端独立，用户端不出现开发者模式。
2. 后台可独立部署（优先独立域名，其次独立端口）。
3. 用户端采用多页面架构。
4. 行程展示采用“地图主视图”方案。
5. 置信度数字不展示给用户。

---

## 4. Non-Goals（本阶段明确不做）

1. 社区内容生产与互动功能。
2. 小红书/其他社交平台分发。
3. 交易闭环（门票、酒店、机票下单）。
4. 多城市联程与跨境行程。
5. 复杂多人协同编辑。
6. Web 端深度离线能力。

---

## 5. 端与权限边界

## 5.1 用户前台（User App）

推荐地址：`trip.example.com`

允许能力：

1. 对话采集需求并生成行程。
2. 查看地图与时间线、局部重规划。
3. 保存、查看、二次编辑历史行程。

禁止展示：

1. API 地址、key、secret、provider、fallback_mode、confidence、debug 输出。
2. 任何 snake_case 字段名或内部枚举值。

## 5.2 后台控制台（Admin Console）

推荐地址：`ops.trip.example.com`（或独立端口 `:5600`）

包含能力：

1. 系统配置（模型、地图、策略开关）。
2. 运行观测（错误、时延、埋点、告警）。
3. 运营管理（反馈查看、活动配置）。

权限策略：

1. 仅 `ADMIN` 可访问。
2. 后台接口与用户接口权限隔离。
3. 用户端不提供后台入口链接。

---

## 6. 信息架构（多页面）

1. `/` 首页：发现与启动  
2. `/plan/new` 对话规划页：单任务采集  
3. `/trip/:id` 行程详情页：地图主视图 + 时间线  
4. `/trips` 我的行程页：历史管理  
5. `/settings` 用户设置页：仅用户偏好，不含开发配置

---

## 7. 用户旅程（主路径）

`首页` -> `开始规划` -> `对话采集` -> `自动生成` -> `行程详情（地图）` -> `保存` -> `我的行程` -> `二次编辑`

---

## 8. 页面规格与 DoD（Definition of Done）

## 8.1 首页 `/`

页面目标：

1. 让用户在 5 秒内理解产品价值并点击开始。

必须有：

1. 主标题 + 副标题 + 主 CTA（开始规划）。
2. 3-6 个场景化示例卡。
3. 最近一次行程入口。

禁止有：

1. 任何工程参数、调试内容、技术状态字段。

DoD：

1. 首屏功能区块不超过 5 个。
2. CTA 可在 1 次点击内进入 `/plan/new`。

## 8.2 对话规划页 `/plan/new`

页面目标：

1. 在不填复杂表单前提下完成需求采集并触发生成。

必须有：

1. 大聊天区（主内容）。
2. 底部固定输入区（回车发送）。
3. 已确认条件摘要（自然语言）。
4. 自动生成与手动生成兜底。

禁止有：

1. 字段名（如 `origin_city`）。
2. 模型模式、fallback 技术字样。

DoD：

1. 首条输入后流程可继续，不会卡死。
2. 失败状态有“重试/继续”明确动作。

## 8.3 行程详情页 `/trip/:id`

页面目标：

1. 让用户可直接执行路线并快速调整。

必须有：

1. 地图主视图（桌面建议 60% 宽）。
2. 时间线与地图双向联动。
3. 局部重规划入口（按天）。
4. 关键提醒（天气/闭馆）自然语言展示。

禁止有：

1. provider/confidence/internal id 等参数字段。

DoD：

1. 点击时间块可高亮地图点。
2. 交通段可打开导航链接。

## 8.4 我的行程 `/trips`

页面目标：

1. 支持历史复用与二次编辑。

必须有：

1. 卡片化历史列表。
2. 搜索与筛选。
3. 继续编辑、复制、删除操作。

禁止有：

1. “加载选中/删除选中”这类后台批量文案。

DoD：

1. 30 条历史下仍保持可读与可操作。

## 8.5 用户设置 `/settings`

页面目标：

1. 管理用户偏好，不承载系统配置。

必须有：

1. 预算偏好、节奏偏好、提醒偏好。

禁止有：

1. 开发配置与调试入口。

DoD：

1. 设置变更可影响后续默认建议。

---

## 9. 状态设计矩阵（每页必补）

每个页面必须覆盖以下状态并有文案与动作：

1. 空状态：告诉用户“下一步做什么”。
2. 加载中：骨架屏或轻量 loading，避免白屏。
3. 失败状态：错误解释 + 主动作（重试）。
4. 网络慢：给等待反馈与可取消策略。
5. 无权限：清晰引导返回或重新登录。

---

## 10. 文案规范（去参数化）

## 10.1 禁用词（用户端）

1. `provider`
2. `fallback_mode`
3. `confidence`
4. `origin_city` / `must_go` 等字段名
5. 直接枚举值：`low|medium|high`, `relaxed|compact`

## 10.2 替换示例

1. `budget_level=medium` -> `预算：中等`
2. `pace=relaxed` -> `节奏：松弛`
3. `fallback_mode=rules` -> `已切换备用方案，继续为你规划`

## 10.3 错误文案规则

1. 先说明影响，再给下一步动作。
2. 禁止直接抛技术异常栈到用户层。

---

## 11. API 映射（页面到接口）

## 11.1 首页 `/`

1. `GET /api/v1/plans/saved?limit=...`：拉最近行程（用于“继续上次”）。

## 11.2 对话规划页 `/plan/new`

1. `POST /api/v1/chat/intake/next`：对话采集下一轮。
2. `POST /api/v1/plans/generate`：触发首版生成。

失败回退：

1. intake 失败 -> 本地规则兜底继续采集（不暴露技术细节）。

## 11.3 行程详情页 `/trip/:id`

1. `GET /api/v1/plans/saved/{id}`：获取行程详情。
2. `POST /api/v1/plans/replan`：局部重规划。
3. `POST /api/v1/plans/save`：保存当前版本。

## 11.4 我的行程 `/trips`

1. `GET /api/v1/plans/saved`
2. `DELETE /api/v1/plans/saved/{id}`
3. `GET /api/v1/plans/saved/{id}/summary`

## 12. 埋点与指标口径

| 事件名 | 触发时机 | 必填字段 | 指标归属 |
|---|---|---|---|
| `home_started` | 首页点击开始规划 | source, user_id | 首页启动转化 |
| `chat_started` | 进入对话并发送首条 | user_id | 对话启动率 |
| `chat_turn_submitted` | 每次发送消息 | turn_index | 对话轮次 |
| `chat_completed` | 采集完成可生成 | duration_ms, turn_count | 采集完成率 |
| `plan_requested` | 触发生成 | source | 生成请求量 |
| `plan_generated` | 生成成功 | duration_ms | 生成成功率/时延 |
| `replan_triggered` | 发起重规划 | scope | 重规划触发率 |
| `plan_saved` | 保存成功 | plan_id | 保存率 |
| `trip_revisited` | 打开历史行程详情 | plan_id | 复访率 |

口径说明：

1. 指标看板统一按日/周聚合。
2. 用户端埋点 payload 禁止携带技术字段字符串到 UI 文案链路。

---

## 13. 视觉密度基线（硬规则）

1. 最大内容宽度：`1200px`，主阅读列 `680-760px`。
2. 间距系统：`8/12/16/24/32`。
3. 首屏不超过 5 个功能块。
4. 每屏仅 1 个主按钮（Primary）。
5. 用户端避免同屏出现 2 个以上可编辑大表单。

---

## 14. 发布、灰度与回滚策略

发布顺序：

1. 先上后台端骨架（不对用户暴露）。
2. 再上用户端多页面路由。
3. 最后上文案去参数化与地图主视图。

灰度策略：

1. 10% -> 30% -> 100% 逐步放量。
2. 每阶段观察核心指标 24h。

回滚条件：

1. 生成成功率下降超过 5%。
2. 对话完成率下降超过 8%。
3. 前端 JS 异常率超过阈值（按 Sentry/日志定义）。

回滚动作：

1. 路由级回滚到旧前端版本。
2. 保留后端接口兼容，避免数据损坏。

## 15. 实施计划（可直接排期）

## Phase A（3-4 天）

1. 建立用户端多页面路由骨架。
2. 建立独立后台端 `apps/admin-console` 骨架。
3. 用户端删除工程参数入口。

## Phase B（4-5 天）

1. 重做 `/plan/new` 全屏聊天体验。
2. 重做 `/trip/:id` 地图主视图 + 时间线联动。
3. 重做 `/trips` 卡片化历史页。

## Phase C（2-3 天）

1. 全量文案去参数化。
2. 状态矩阵补齐（空/错/加载/慢网）。
3. 埋点校验与看板联调。

---

## 16. 最终验收清单（上线门槛）

1. 用户端任一页面不出现开发者入口。  
2. 用户端不出现 provider/confidence/字段名。  
3. 行程详情页地图可作为主视图并稳定联动。  
4. 用户不进入后台也可完整完成规划链路。  
5. 后台端仅 ADMIN 可访问且独立部署。  
6. 核心指标达到灰度门槛后再全量。

---

3. 任何新增页面必须补 DoD、状态矩阵和埋点映射。
