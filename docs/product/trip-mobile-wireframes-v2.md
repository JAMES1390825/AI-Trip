## 1. 文档目标

这份文档定义移动端新版主链路的页面结构与状态流，目标不是单纯“换一套更像竞品的 UI”，而是将现有产品能力重组为更适合手机端的单任务流程：

`轻采集 -> AI 生成 -> 地图执行 -> 现场调整 -> 保存/分享`

重点解决 4 个问题：

1. 当前规划入口信息密度偏高，用户不容易快速开始。
2. 生成态缺少过程感，等待期间价值感不足。
3. 结果页虽然信息完整，但“地图”和“执行动作”还没有成为绝对主轴。
4. 点位详情更多是内容展示，尚未充分承担“现场决策”职责。

---

## 2. 设计原则

### 2.1 借鉴什么

1. 极简采集页：先收最少必要信息，保证用户能开始。
2. 地图主视图：路线、点位、日期联动是移动端核心容器。
3. Bottom sheet 交互：详情和操作不离开当前上下文。
4. 生成过程可视化：让用户感知 AI 在逐步构建路线。

### 2.2 我们不照搬什么

1. 不把所有资产能力都堆到结果页首屏。
2. 不把编辑能力藏得很深，只做“看图”不做“改图”。
3. 不只给景点介绍，要补足执行信息与风险提示。
4. 不做跨天信息大爆炸式地图，让用户失去当下判断能力。

### 2.3 我们的强化点

1. 地图上每个点都要有动作：导航、锁定、替换、问 AI。
2. 行中态优先：默认帮助用户判断“今天接下来去哪”。
3. 推荐必须可解释：为什么排这个点、风险是什么、替代是什么。
4. 所有失败态都要可恢复：地图异常、定位失败、无坐标、生成失败都有下一步。

---

## 3. 信息架构

底部导航收口为两个核心页面，避免入口重复和职责重叠。

1. `行程`
2. `AI规划`

新版关系：

- `行程`：已保存行程、提醒、分享与回看入口。
- `AI规划`：承接从输入到生成到结果的主流程，并直接填写规划偏好。

---

## 4. 主流程

主流程：

`AI规划入口 -> 规划输入页 -> 目的地搜索页 -> 日期选择 sheet -> AI 生成中 -> 地图结果页 -> 点位详情 sheet -> 保存/分享 -> 行中优化`

辅助流程：

1. `行程` 中打开已保存路线时，直接进入地图结果页。
2. 生成失败时可停留在输入页并保留用户已填数据。
3. 地图异常时可降级到纯时间线模式。

## 5. 页面一：规划输入页 `/map/plan-entry`

### 5.1 页面职责

这是移动端新主入口，只做“够用的采集”，让用户 10 秒内开始规划。

### 5.2 页面字段

首屏直接收齐规划所需核心信息：

1. 目的地
2. 日期或天数
3. 偏好标签
4. 预算
5. 节奏
6. 主动作选择：`发起 AI 行程` / `自己规划`

### 5.3 低保真线框

```text
+--------------------------------------+
| [返回]                               |
+--------------------------------------+
| 你想去哪里？                         |
| [ 搜索目的地 / 城市 / 景点关键词 ]    |
+--------------------------------------+
| 你想去多久？                         |
| [ 开始日期 - 结束日期 ]              |
+--------------------------------------+
| 旅行偏好                             |
| [经典必玩] [吃吃喝喝] [小众探索]      |
| [拍照出片] [逛街购物] [citywalk]     |
| [自然风光] [文艺展览] [历史古建]      |
+--------------------------------------+
| 规划偏好                             |
| 预算 [低预算] [中预算] [高体验]      |
| 节奏 [轻松] [紧凑]                  |
|--------------------------------------|
| [ 发起 AI 行程 ]                     |
| [ 自己规划 ]                         |
+--------------------------------------+
```

### 5.4 交互规则

1. 目的地点击后进入独立搜索页，而不是原地用复杂下拉。
2. 日期选择使用 bottom sheet 日历，不跳全页。
3. 标签默认多选，最多高亮 3 个推荐标签。
4. 只有目的地是强必填；未选日期时允许用“灵活天数”生成草案。

### 5.5 页面说明

1. 不展示技术性表单字段名，不出现 JSON 感配置项。
2. 预算和节奏直接在 AI 规划入口填写，不再拆到独立偏好页。
3. 只保留两个主 CTA，避免第三种模糊中间态按钮。
4. `自己规划` 不是放弃 AI，而是进入可手动搭点的轻编辑模式。

## 6. 页面二：目的地搜索页 `/map/search-destination`

### 6.1 页面职责

承接目的地输入，优先完成“城市确认”，其次提供联想与热门候选。

### 6.2 低保真线框

```text
+--------------------------------------+

+--------------------------------------+
| 你想去哪里？                         |

|--------------------------------------|

|--------------------------------------|
| 东京都                               |
| 日本东京都                           |
|--------------------------------------|
| ...                                  |
+--------------------------------------+
```

### 6.3 交互规则

1. 搜索结果优先展示城市，再展示景点或行政区。
2. 选择后直接返回输入页，并填充标准化城市名。
3. 输入清空后展示历史搜索和热门城市。

1. 搜索结果需要标准化字段：`name`, `display_name`, `country`, `region`, `city_code`, `lat`, `lng`。
2. 必须区分“城市级目的地”和“POI 级关键词”，避免误把景点当行程城市。

## 8. 页面四：AI 生成中 `/map/generating`

### 8.1 页面职责

让等待过程具备“正在生成可执行路线”的感知，而不是单纯转圈。

### 8.2 低保真线框

```text
+--------------------------------------+
|           地图底图 + 路线动画         |
|   DAY 1 / DAY 2 marker 逐步出现       |
|                                      |
|--------------------------------------|
| DAY 2 什刹海胡同与恭王府风情          |
| 1. 恭王府博物馆                       |
| 2. 什刹海                             |
| 3. 烟袋斜街                           |
|                                      |
| 正在规划第二天（AI生成）              |
|                                      |
|               [取消]                 |
+--------------------------------------+
```

### 8.3 生成态文案建议

不要只显示“生成中”，应显示对用户有价值的过程文案：

1. 正在压缩跨区通勤
2. 正在避开闭馆时间
3. 正在补齐午餐与休息节点
4. 正在校验地理顺序

### 8.4 交互规则

1. 若 1 秒内返回，可直接跳过动画。
2. 若超过 1 秒，进入生成中页并展示阶段文案。
3. 用户可取消，取消后回到输入页并保留已填数据。
4. 生成失败时给出“重试生成”和“改下条件”两个动作。

### 9.1 页面职责

这是移动端核心页，承担“看路线、看今天、做调整、打开点位详情”的统一容器。

### 9.4 结果页分层建议

首屏默认只保留 3 层内容：

1. 当前日期 tab
2. 当天摘要
3. 当天 POI 列表或总览摘要

以下模块延后或折叠：

1. 便签
2. 行李清单
3. 签证/证件
4. 图片空间

1. 地图点击 marker，bottom sheet 自动滚动到对应 POI。
2. 切换日期 tab，地图只高亮当日路线与点位。
3. 提供 `总览 / 今天` 切换，帮助旅途中用户快速聚焦当前天。
4. `AI 优化` 默认优化当前天，二次确认后才跨天调整。
5. `编辑路线` 进入轻编辑态，支持换点、插点、删除、锁定。

## 14. 状态流拆解

## 14.1 页面状态机

```text
空白输入
  -> 输入目的地
  -> 选择日期/标签
  -> 发起生成
  -> 生成中
  -> 生成成功
  -> 地图总览
  -> 单天详情
  -> 点位详情
  -> 快速优化
  -> 保存/分享
```

异常分支：

```text
输入不足 -> 提示补齐
生成失败 -> 保留原输入 + 提供重试
地图异常 -> 降级纯列表模式
坐标不足 -> 显示风险卡 + 允许继续浏览
分享失败 -> 保留已保存行程 + 提示重试
```

## 14.2 首屏默认策略

1. 首次生成成功后默认进入 `总览`。
2. 若当前日期落在行程范围内，优先推荐切到 `今天`。
3. 从“已保存行程”再次打开时，优先恢复上次查看的日期 tab。

## 14.3 权限与只读模式

1. 分享链接打开后进入只读模式。
2. 只读模式下保留：地图、时间线、导航、提醒。
3. 只读模式下关闭：优化、编辑、删除、回退。

## 18. 验收标准

1. 新用户在 10 秒内能完成首次规划提交。
2. 生成等待超过 1 秒时，用户能看到明确过程感页面。
3. 地图结果页首屏只呈现当前决策需要的信息，不堆叠资产模块。
4. 用户点击任一点位后，能在 1 步内完成导航或替换决策。
5. 行中用户能在 30 秒内完成“查看今天 -> 判断下一站 -> 发起优化”。

---

## 19. 决策结论

移动端新版不应再围绕“长表单 + 大一统规划页”组织，而应围绕 4 个连续场景组织：

1. 快速开始
2. 感知 AI 正在生成
3. 在地图上执行
4. 在现场轻调整

这样既能借鉴成熟竞品的交互路径，也能保留 AI Trip 自己的差异化重点：更可执行、更可解释、更容易在旅途中继续用。
